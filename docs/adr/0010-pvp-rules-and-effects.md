# 0010. 対戦(PvP)のゲームルールと効果モデル

- ステータス: 承認(設計決定。実装は後続 PR で段階的に行う)
- 日付: 2026-06-14
- 関連: ADR 0001(正確性ゲートのダメージ)を拡張、ADR 0007(先行入力)を踏襲、
  ADR 0008(状態同期の時間軸/入力軸分割)を踏襲、ADR 0009(PvP 前方設計)を具体化、
  ADR 0011(対戦ネットコード)と対をなす

## 文脈

ADR 0009 で PvP の技術的足場(決定論・seed 共有・コマンドログ・serialize/restore・
snapshot 語彙の一般化)を予約したが、「対戦とは何か」というゲームルール自体は
未確定だった。CONTEXT.md の対戦モードは「オンラインで他のプレイヤーと同時にプレイし
勝敗を決める」とだけ定義し、相手プレイヤー・勝敗条件・カード効果の中身を持たない。

加えて矛盾があった。cards.ts のカードは `damage` しか持たないのに、カード名
(氷牢・地淵)は効果を匂わせる。一方 ADR 0001 は「ダメージはカード固定、速度は
先撃ちの読み合いにのみ効く」と決めている。対戦の相互作用を設計するには、この空白と
矛盾を埋める必要があった。

各分岐で検討した代替案:

- **相互作用**: 並走レース(干渉なし)/ 直接ダメージ(相互 HP)/ 状態効果あり
- **効果の作用先**: 相手の打鍵への干渉 / 相手のゲーム経済 / 自己バフ
- **勝敗**: 相互 HP のみ / 相互 HP + 制限時間 / 残ダメージレース
- **効果の時間**: 瞬間のみ / 時限持続 / 経時(DoT/HoT)
- **効果の表現**: 宣言的データ(判別共用体)/ コード(クロージャ)/ 固定 enum
- **効果の通貨**: 専用ユーティリティカード / お題長(詠唱時間)/ ダメージとの直接トレード

## 決定

### ゲームルール

1. **状態効果あり型**。2 人は固定ダメージに加え、カード効果で影響し合う。
2. **効果はゲーム経済と自己にのみ作用し、相手の打鍵には一切干渉しない**。凍結=
   「打てなくする」のような入力干渉は採らない。理由: (a) サーバーは相手に誤入力を
   強制できず、ADR 0009 の決定論 + サーバー権威と衝突する、(b) タイピング実力と
   無関係に事故り、本作の核(タイピング実力 × デッキ戦略)を薄める。
3. **勝敗 = 相互 HP + 制限時間**。両者が HP を持ち、先に相手を 0 にした側が勝つ。
   決着しないまま制限時間が尽きたら残 HP の多い側が勝つ。
4. **効果は時限持続(経時 DoT/HoT は採らない)**。持続効果の失効は権威時刻 `atMs` の
   遅延評価で解決し、HP を毎 tick 増減させるループは作らない。HP は発動イベントでのみ
   変わる(ADR 0008 のイベント駆動・rAF 撤廃を維持)。表示のカウントダウンは既存の
   時間 tick(約 100ms / `snapshotTimers`)で足りる。効果失効はこの遅延評価(参照時に読む値)で
   足り、能動発火(タイマ)は不要。一方、時間切れの勝敗確定はどちらも打鍵しなくても起きる権威
   イベントなので、表示 tick ではなく DO の権威タイマで能動発火する(発火主体は ADR 0011 #10)。

### 効果モデル

5. **効果は宣言的データ(判別共用体)で表す**。`Card` に `effects: Effect[]` を
   持たせ、`Effect` は `{ kind, ...params }` の判別共用体とする。エンジンは kind ごとの
   ハンドラで適用する。理由: `BattleEvent` と同じ既存規約、ADR 0004 の
   「関数と定数中心」、ADR 0009 の決定論・serialize/restore・ワイヤ送信・サーバー検証の
   すべてに乗る。コード(クロージャ)はシリアライズ不可・サーバー検証困難で却下。
6. **効果の通貨はお題長(詠唱時間)**。効果付きカードは読みを長くし、かつ
   damage/キー効率を純攻撃カードの曲線より下げる。「効果が欲しければ詠唱に時間を払い、
   その間の先撃ちリスクを負う」形にし、ADR 0001(速度 = 先撃ち)と整合させる。
7. **派生制約: 効果はダメージの数値そのものに触れない**。「次の攻撃 +X」
   「相手のダメージ半減」は禁止。効果が操作してよいのは HP 軸・テンポ軸・カード経済軸の
   3 つに限る。これで ADR 0001 の「ダメージはカード固定」を不変に保つ。
8. **v1 効果メニュー(3 軸)**:

   | 軸     | kind      | 意味                                                       | 解決契機               |
   | ------ | --------- | ---------------------------------------------------------- | ---------------------- |
   | HP     | `heal`    | 自陣 HP +amount(最大 HP 上限)                            | 即時                   |
   | HP     | `shield`  | 自陣にシールド +amount。被弾でダメージを肩代わりして消費   | 被弾まで(+任意で時限) |
   | テンポ | `haste`   | 自分の CD を T 秒間 −ms                                     | 時限持続               |
   | テンポ | `slow`    | 相手の CD を T 秒間 +ms                                     | 時限持続               |
   | 経済   | `discard` | 相手の手札をランダムに 1 枚捨てさせ、山札から補充          | 即時                   |
   | 経済   | `sift`    | 自分の山札操作(上を見て並べ替え / 1 枚先引き)            | 即時                   |

   `discard`/`sift` は山札に触れるが、山札はサーバー権威の seed 駆動(ADR 0011)で
   あり決定論は保たれる。

9. **スタック規則 = リフレッシュ(上書き)**。同種の時限効果(`haste`/`slow`)を
   再付与したら持続時間を上書きし(延長しない)、効果値は最大を採る。`shield` のみ
   上限付きで加算する。理由: カスタムデッキ(ADR 0011)+ 経済軸ありでコンボが
   暴走しやすく、加算スタックは `slow` 重ねがけによる相手の完全拘束を招く。
   リフレッシュは上限が自明で決定論・シリアライズに優しい。

### バランス定数(別途のバランス分析が根拠)

10. **プレイヤー HP = 80、制限時間 = 120 秒**。`time ≈ HP / 実効DPS` で結合するため、
    HP を変えたら制限時間もスケールさせる。
11. **回復は ≤ 7HP/詠唱**(デッキ平均ダメージ 8.7 未満)を不変条件とする。亀(回復固め)の
    停滞しきい値 `f* = 平均ダメージ / 回復量` は制限時間に依存しないため、亀は
    制限時間ではなく回復量で封じる。
12. 補足: 人間のプレイ域(3〜8kps)では全カードがタイピング律速で、1500ms クールダウンは
    先行入力(ADR 0007)に隠れて試合長にほぼ効かない。試合長は HP と打鍵速度で決まる。
    厳密には「1 発動 / 1.5 秒」のスループット天井は存在するが、律速になるのは詠唱時間が
    1500ms を下回る超人速度のみ(最短 wave で約 12.7kps、最長 abyss で約 32kps)。人間域では
    `cycle = max(詠唱時間, 1500ms) = 詠唱時間` となり天井に当たらない。

### 効果・勝敗の解決規約(設計レビュー反映で確定)

PR レビューで洗い出した「実装で詰まる未定義事項」を以下に確定する。

13. **haste/slow は「次回 CD のみ」方式 + 有効窓**。進行中のクールダウン残量は書き換えず、効果適用後に
    新規開始するクールダウンにのみ ±ms を適用する(進行中 CD を後付けで延ばすと、相手の効果到着が
    自陣のローカル受理タイムライン=先行入力の再スタンプ時刻 ADR 0007 に侵入して巻き戻しが要るため)。
    `durationMs` の窓内に開始する各 CD に適用する(窓 = 効果有効期間。窓終端は一時停止で ADR 0011 #11 の
    オフセットを受ける)。
    有効性の正直な限界: #12 のとおり人間域では `cycle = 詠唱時間 > CD` なので、+ms 後も
    `CD + slow < 詠唱時間` の限り CD は律速にならず、テンポ軸は短詠唱カード / 高 kps 域(詠唱時間が
    CD に接近する局面)と先行入力バッファ回転でのみ効く補助軸である。中央値域で死に効果なら、fallback
    として CD ではなく先行入力のドレイン時刻に作用させる案を別途検討する(本 ADR では CD 作用に留める)。
14. **ダメージ解決パイプラインと HP/シールド上限**。被弾は固定順で解決する: ①shield から控除 →
    ②貫通分を HP から控除 → ③HP ≤ 0 を記録。**勝敗確定(④)は同一権威 atMs の全発動・全効果を適用
    しきった後に一括評価する**(発動ごとに即終了させない。これにより #16 の引き分けが成立する)。最大 HP は
    初期 HP(= 80)とし、heal の超過分は破棄する(クランプ)。shield は加算上限(暫定: 平均ダメージ
    1〜2 発分)を設け、時限失効時は残量を破棄する。shield は `card.damage` に介入せず HP 前段の吸収
    バッファであり、ADR 0001 のダメージ固定を侵さない。
15. **効果適用順序(決定論の要)**。同一発動内の複数効果は「ダメージ → 自陣バフ(heal/shield/haste/
    sift)→ 相手デバフ(slow/discard)」の固定順で適用する。両陣営が同一権威 atMs で発動した場合は
    playerId の安定ソートで全順序化し、その後 #14 ④の一括勝敗判定を行う。これで `effects[]` の評価が
    一意になり「同じ入力列 → 同じ状態」(ADR 0009 #1)を満たす。
16. **勝敗裁定のタイブレーク**。atMs が異なれば #3 のとおり先に相手を 0 にした側が勝つ。同一権威 atMs
    で両者の HP が 0 以下になった場合(#14 の一括評価時点で双方 ≤ 0)、および制限時間切れで残 HP が
    同値の場合は引き分け(draw)とする。
17. **discard の枯渇フォールバック**。相手の山札・捨て札がともに空で補充できない場合、discard は no-op
    とする(例外を投げない)。捨てさせた札は相手の捨て札へ送る。詠唱中(発動前)のカードは discard の
    対象外とする。

### 純攻撃カードのバランス是正(ray/abyss)

本 ADR の「純攻撃 10 種は据え置き」方針の**明示的な例外**として、ray と abyss の 2 種のみ
damage を引き上げる(ray 10→11、abyss 16→17)。読み・displayText・cooldownMs・他 8 種は据え置く。

18. **是正理由(打鍵あたり効率の単調性回復)**。効率を damage/打鍵数(最短ローマ字路)で測ると、
    現状は ray が谷(0.286 < thunder 0.290)になり「長いほど効率が高い」非線形リターン(ADR 0001)が
    反転していた。さらに abyss が meteor と同値(ともに 0.333)で頭打ちになり「最長 = 最高効率」が
    崩れていた。ray 10→11(0.314)・abyss 16→17(0.354)で谷と頭打ちを解消し、読み長順の狭義単調増加を
    回復する。effects 軸(#5〜#9)・ダメージ固定規約(#7)・Effect 型・MatchEngine には触れない。
19. **デッキ平均への影響は軽微**。デッキ平均ダメージは 8.7→8.9 とほぼ不変で、HP 80 / 制限時間 120 秒
    (#10)や回復 ≤7HP(#11)の不変条件を揺るがさない。よって本是正はバランス定数の再導出を要しない
    局所調整である。

## 結果

### 良い点

- 「対戦とは何か」が固定ダメージ + 3 軸の効果として明確になり、cards.ts の `damage`
  のみという空白と、効果を匂わせるカード名との矛盾が解消される。
- 効果が打鍵に干渉せずゲーム経済のみを操作するため、サーバー権威で決定論的に検証でき
  (ADR 0011)、本作の核を薄めない。
- ADR 0001 を否定せず拡張する形に収まり(ダメージ固定は不変、通貨はお題長)、既存の
  カード経済設計をやり直さずに済む。
- 効果を宣言的データにしたことで、ワイヤ送信・serialize/restore・サーバー検証が
  素直に実装できる。

### トレードオフ

- 効果を既存攻撃カードと同じ「お題長」軸で払うため、効果付きカードはお題長 = 強さの
  曲線からサブ曲線に外す調整が要り、バランス調整がカードごとに増える。
- カード経済軸(`discard`/`sift`)はカスタムデッキでコンボが読みにくい。スタックの
  リフレッシュ化と値の抑制で封じる方針だが、実測でのチューニングが前提。
- バランス定数(HP 80 / 120 秒 / 回復 ≤7)は誤入力ゼロ前提の解析が根拠であり、
  実プレイでの再調整余地がある。
