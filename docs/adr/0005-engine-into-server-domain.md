# 0005. 判定エンジンをサーバーのドメイン層へ移設し、web は subpath export で借用する

- ステータス: 承認
- 日付: 2026-06-12
- 関連: ADR 0002(純TSエンジン分離)、ADR 0004(サーバー DDD 構成)
  — 両 ADR の「engine は独立した共有パッケージ」という配置の記述を本 ADR が上書きする。
  フレームワーク非依存の純 TS という性質、および「判定ロジックを二重化しない」
  という原則は引き続き有効。

## 文脈

ADR 0002 で分離した `packages/engine` は、web と server が対等に依存する
中立な共有パッケージとして置かれていた。しかし:

- 対戦モードではサーバーが判定の権威になる構想であり(ADR 0004)、
  「誰がエンジンを所有するのか」を中立パッケージという構図は曖昧にする。
  実態は「サーバーのドメインそのもの」であり、web はそれをローカルで
  実行させてもらっている側である。
- `packages/` 階層の住人は engine ただ一つで、apps/ と packages/ の
  二階層を維持するコストが構造上の利得に見合っていない。

なお ADR 0002 が挙げた利点(UI なしでの網羅テスト、サーバーへの持ち込み、
ロジックの無傷性)は、エンジンの置き場所には依存しない。本決定は利点の
獲得ではなく、所有権の明確化と構造の単純化である。

## 決定

`packages/engine` を `apps/server/src/domain/engine/` へ移設する(テスト含む)。

- **所有権**: 判定エンジンはサーバーのドメイン層の一部とする。
  ソロモードは「server のドメインコードを web がローカル実行している」
  という借用の関係であり、オフライン完結という性質は変わらない。
- **web への公開**: `@magic/server` の `package.json` に subpath export
  `"./engine": "./src/domain/engine/index.ts"` を定義し、web は
  `@magic/server/engine` を import する。web に公開するのはこの一点のみとし、
  将来 `domain/` 直下に生まれるサーバー固有ロジック(マッチング、ルーム管理等)
  は export しない。境界は package exports で API として強制する。
- **境界の維持**: engine は `domain/` 直下に溶かさず `domain/engine/`
  サブディレクトリとして境界を保つ。ADR 0004 の依存ルール
  (domain は Hono にもランタイム API にも依存しない純 TS、Vitest でテスト)
  は engine にそのまま適用される。
- **抜け殻の処理**: 空になった `packages/` ディレクトリと
  `pnpm-workspace.yaml` の `packages/*` glob は削除する。workspace は
  `apps/*` のみとし、共有パッケージが再び必要になったら glob を戻す。

## 結果

### 良い点

- 「判定の権威はサーバー」という対戦モードの構想が、構造(コードの住所)
  と一致する。
- web から import できる範囲が subpath export で明示され、サーバー内部への
  依存が API として制限される。
- workspace が apps/ 一階層になり、実在するものだけを語る構造になる。

### トレードオフ

- web(フロントエンドアプリ)が `@magic/server`(バックエンドアプリ)に
  依存するという珍しい依存方向が生まれる。実際に共有されるのは純 TS の
  engine のみだが、初見の読者には説明が要る(本 ADR がその説明である)。
- server の `package.json` が「Workers にデプロイされるアプリ」と
  「web へ公開するライブラリ」の二役を持つ。
- engine の破壊的変更が web に直接波及する点は従来どおりだが、
  変更がサーバーアプリの内部ディレクトリで起こるため、web への影響を
  見落としやすくなる。レビューで `domain/engine/` の変更時は web 側の
  利用箇所を確認する運用とする。
