# 0006. web のディレクトリを pages / ui / lib の責務分離レイヤーで構成する

- ステータス: 承認
- 日付: 2026-06-12
- 関連: ADR 0002(純TSエンジン分離)、ADR 0005(エンジンのサーバー移設)
- 参考: ZOZOTOWN フロントエンドのディレクトリ設計
  (https://techblog.zozo.com/entry/zozotown-frontend-directory-design)

## 文脈

web の `src/ui/` はフラットな7ファイル構成で始まったが、デッキ構築画面や
対戦モードの追加を見据えると「新しいファイルをどこに置くか」の判断基準が
存在しない。ZOZOTOWN の責務分離パターン(UI / Models / Pages / Layouts /
Functional の5分類 + 一方向依存)を参考にしたが、そのまま持ち込むと
このプロジェクトの前提と衝突する:

- **Models が存在し得ない**: ZOZO の Models は「ドメインロジックを含む
  コンポーネント」だが、ADR 0002 はドメインロジックを engine に集約し
  コンポーネントを「UI の皮」に徹させると決めている。Models に入る住人は
  定義上存在しない。
- **Functional は React のイディオム**: 「UI を持たないコンポーネント」は
  Provider 等を想定した分類で、Svelte では `.svelte.ts` モジュールや
  素の TS として書くのが自然であり、コンポーネントの分類としては不要。

採用すべきは5分類という形ではなく、「迷わない分類基準」と「一方向依存」
という設計の核心である。

## 決定

`apps/web/src/` を次の構成とする:

```
src/
├── main.ts
├── App.svelte        ← ルートシェル(唯一、全 pages を参照できる)
├── pages/            ← ルート単位のページとその専用子コンポーネント
│   ├── home/
│   └── game/         ← Game.svelte と Start/Battle/Result の各 Screen
├── ui/               ← ページ横断で再利用する表示部品(例: Card)
└── lib/              ← 非コンポーネントの TS / .svelte.ts(router、API クライアント等)
```

- **Page の単位はルート**: ルーターが解決する単位(`#/home`, `#/game`)を
  ページとする。ページ内遷移の Screen 群はそのページ専用の子コンポーネント
  として同じページフォルダにコロケーションする。トップレベルのページ
  フォルダ同士の参照は禁止し、ページの独立性を保つ。
- **依存は一方向のみ**: pages → ui → lib。逆方向は禁止。
  App.svelte だけが pages を参照できる。
- **ui の engine 依存は型のみ**: `ui/` から engine への `import type` は
  許可するが、判定・計算など engine の関数呼び出しは禁止。関数を呼べるのは
  pages と lib のみ。ui のコンポーネントは判定済みの値を props で受け取る
  (例: Card は「クールダウン中か」を計算せず `disabled` を受け取る)。
- **layouts は作らない**: 現状アプリに共通の視覚的骨格が存在しないため、
  空の分類を先回りで作らない。共通ヘッダー等が実際に生まれた時点で
  `layouts/` を追加する。
- **lib の事前分割もしない**: `lib/` 直下が手狭になってから用途別に割る。

## 結果

### 良い点

- 「特定ページ専用か → pages、ページ横断の表示部品か → ui、
  コンポーネントでないか → lib」の3問で配置が決まり、判断が割れない。
- ADR 0002 の「UI は薄い皮」が依存ルール(ui は型のみ依存)として
  具体化され、ドメインロジックがコンポーネントへ漏れ出すのを防ぐ。
- view-model の二重定義を避けつつ(型は engine から借りる)、
  ロジックの境界は import の種類という機械的に判別できる線で守られる。

### トレードオフ

- 依存ルールは lint で強制せず、レビュー運用で守る。違反コストが
  顕在化したら lint 導入を検討する。
- `ui/` は当面 Card 程度でほぼ空になるが、無理に部品を「汎用化」して
  埋めるより健全と判断した。
- 参考元と分類名が部分的に同じ(ui, pages)だが意味が異なるため、
  ZOZO の記事を知っている読者はかえって混乱しうる(本 ADR がその説明である)。
