# 0009. 対戦モード(PvP)の前方設計 — 決定論エンジンを土台に和解の足場を予約する

- ステータス: 提案(着手は PvP 実装時) — 本 ADR は実装を伴わない
- 日付: 2026-06-14
- 関連: ADR 0002(純TS決定論エンジン)、ADR 0003/0004(Hono + Durable Objects 構想)、
  ADR 0008(バトルの状態同期)

## 文脈

ロードマップに対戦モード(オンライン PvP)があり、WebSocket + Cloudflare Durable Objects +
サーバー権威の判定を構想している。パフォーマンス/アーキテクチャレビューで次が確認された:

- 決定論エンジン(時刻・rng の外部注入)はクライアント予測とサーバー和解(reconciliation)に
  有利な「土台」がある。
- しかし和解を実装する「足場」が無い: 状態の復元機構、受理打鍵(入力)の履歴、seed の保存、
  そして `BattleSnapshot` がソロの「的(targetHp)」語彙のままで相手プレイヤー・権威時刻を
  表現できない。
- 一方で、PvP ルール/ワイヤプロトコル/和解戦略が未確定の今これらを先回り実装すると、
  誤った抽象を公開境界に固定する(=アーキテクチャ版の早すぎる最適化)リスクが高い。

## 決定

**実装はしない。** PvP 着手時の前提・制約を以下に記録するに留める。

1. **決定論性を不変条件とする**: 時刻・rng を外部注入する設計を維持し、和解の前提
   「同じ入力列 + 同じ `atMs` 列 + 同じ seed → 同じ状態」を保証する。
2. **seed 可能な rng**: rng は既に注入可能で、テストは mulberry32 を使う。対戦では seed を
   保存・共有して山札のシャッフルを再現可能にする。
3. **入力をコマンド列として記録**: 受理打鍵を `{ type, handIndex, key, atMs }` のコマンド列として
   完全記録し、`from-snapshot + replay(commands)` で任意 tick を再構成する API を将来追加する。
   現状の `eventLog` は集計(統計)用途であり、コマンドログとは責務を分離する。
4. **serialize / restore**: `BattleEngine` / `TypingSession` に `serialize()`/`restore()` を将来追加する。
   ただし内部 `Candidate` 形状を公開境界に固定しないよう、復元はエンジン内部に閉じる。
5. **snapshot 語彙の一般化**: `BattleSnapshot` の「的(targetHp/targetMaxHp)」は、PvP 着手時に
   自陣/敵陣の `playerState { hp, casting, cooldown }` へ一般化し、権威時刻
   (`serverTick`/`authoritativeAtMs`)を持たせる。ソロは「敵 = サンドバッグ」を同型で表現する。
   型変更はワイヤプロトコルと和解戦略の確定と同時に行う。

## 結果

### 良い点

- レビューとグリルで得た知見が散逸せず、PvP 着手時の手戻りを減らせる。
- 「決定論エンジンは判定の権威の土台」という資産を明文化できる。

### トレードオフ

- あくまで予約であり拘束力は弱い。PvP ルール確定時に内容を修正する可能性がある。
- 5 の snapshot 一般化は ADR 0008 で増えた 2 系統 API(timers/state)をさらに変える可能性がある。
