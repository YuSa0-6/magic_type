# 0004. サーバー(Hono)の層構成と単一ドメインパッケージ

- ステータス: 承認
- 日付: 2026-06-12
- 関連: ADR 0002(純TSエンジン分離)、ADR 0003(Hono + Cloudflare Workers)
- 参考: [Hono マイベストプラクティス](https://zenn.dev/yodaka/articles/ad49f29a54ceba)

## 文脈

ADR 0003 で Hono + Cloudflare Workers のサーバーを立てたが、現状の API は
`/api/health` のみで、`apps/server/src/index.ts` にルートを直書きしている。
対戦モード(オンラインPvP)ではマッチング・ルーム管理・サーバー権威の判定が
加わる構想があり、その際に場当たり的にルートへロジックを書き足すと
テスト不能な Worker コードが育つリスクがある。先に層の構成と依存ルールを
決めておきたい。

前提として、このリポジトリには既にドメイン層が存在する。ADR 0002 で分離した
`packages/engine` がそれであり、タイピング判定・バトルルールの唯一の正である。

当初は「ゲームルールは engine、サーバー固有の概念(ルーム・マッチング等)は
`apps/server/src/domain`」という分担を検討したが、この線引きには
ドメイン欠乏症(anemic domain model)の危険がある。対戦モードにおいて
「ルーム」と「バトル進行」は本来ひとつのモデルであり、境界を先に引くと
サーバー側ドメインが振る舞いを持たないデータの入れ物と手続きに痩せていく。

検討した構成:

- **A. 教科書的クリーンアーキテクチャ4層**
  (domain / application / infrastructure / presentation + composition root):
  依存ルールは最も厳格だが、ポート定義と DI の定型コードが多く、
  現規模には過剰
- **B. Hono コミュニティの実践型レイアウト + サーバー内 domain**(参考記事):
  domain / lib / routes / middleware / schema / utils。実用的だが、
  engine とサーバー内 domain の二重ドメインになり上記の欠乏症リスクを抱える
- **C. B のレイアウトから domain を外し、ドメインを単一パッケージに統合**:
  `packages/engine` を製品全体のドメイン層と再定義し、サーバー発の概念も
  ここにモデリングする

## 決定

**C を採用する。** ドメインの正は `packages/engine` ただ一つとし、
`apps/server/src` は次の層で構成する:

```
packages/engine/    # 製品全体のドメイン層(唯一の正)。タイピング判定・バトルに加え、
                    # ルーム・マッチング等のサーバー発の概念も将来ここにモデリングする

apps/server/src/
  lib/              # 外部 SDK・ランタイム依存の薄いラッパー(将来: D1 / KV / DO 等)
  routes/           # 機能ごとの Hono サブルーター(HTTP の関心事のみ)
  middleware/       # ロガー・認証等(必要になったら)
  schema/           # Zod によるリクエスト/レスポンススキーマ(リソースごとに分割)
  utils/            # 汎用関数
  index.ts          # エントリポイント(サブルーターのマウントと app の export)
```

`apps/server/src/domain` は**作らない**。engine とサーバー内ドメインの
区別そのものを無くすことで、ロジックの置き場の迷いと二重化を根絶する。

### 依存ルール

- `routes` → `@magic/engine` の一方向。engine はどのアプリにも依存しない
- ビジネスルール(判定・進行・マッチング条件等)は engine に書く。
  routes は「HTTP ⇄ ドメイン」の変換(取り出し・検証・整形)に徹し、
  ルールを持たない
- engine は Hono にも Cloudflare Workers のランタイム API にも依存しない
  純 TypeScript を維持する(ADR 0002 のとおり)。ランタイム依存が必要な
  処理は server の `lib` に置き、engine へは値・コールバックとして渡す
- 各ルートファイルは自分の `Hono` インスタンスを作って export し、
  `index.ts` が `app.route()` でマウントする(Hono 公式の
  "Building a larger application" パターン)

### パッケージ名について

`@magic/engine` の名前は維持する(import の波及を避ける)。本 ADR をもって
engine の役割を「タイピング/バトルのエンジン」から「製品全体のドメイン層」へ
再定義する。クライアントに不要なサーバー発のモジュールが増えてバンドルが
問題になった場合は、エントリポイント分割(subpath exports)で対応し、
パッケージ分割は最後の手段とする。

### 適用タイミング

本 ADR と同じ PR で `routes/` を導入し、`/api/health` をサブルーターへ
載せ替えて動く見本にする。`middleware` / `schema` / `lib` / `utils` などの
ディレクトリは中身が生まれるときに作り、空フォルダは置かない。
ルーム・マッチング等の engine への追加は対戦モード着手時に行う。

## 結果

### 良い点

- ドメインモデルがひとつに保たれ、「どちらに書くか」の判断と二重化が消える。
  ルームとバトル進行を同じモデルとして振る舞い中心に育てられる
- engine が純 TS のため、Workers ランタイムなしで Vitest によりテストできる
  (ADR 0002 と同じ利点をサーバー側でも得る)
- クライアントとサーバーが同じドメインを参照するため、対戦モードの
  サーバー権威判定とクライアント表示の不一致が構造的に起きにくい
- A 案に比べ定型コード(ポート定義・DI の組み立て)が少なく、現規模に見合う

### トレードオフ

- engine にサーバーでしか使わない概念が同居する。クライアントのバンドルへの
  混入は subpath exports で防ぐ運用が必要になり得る
- 強制力のある依存逆転(interface 越しの注入)はないため、「ルールは engine、
  routes は変換のみ」の規律はレビューで守る運用になる
- engine の変更がクライアント・サーバー双方に波及するため、破壊的変更の
  影響範囲は常に両方を意識する必要がある
- 将来ユースケース層が必要な複雑さ(トランザクション境界、複数ストレージの
  協調等)が出てきた場合は、application 層の追加を別 ADR で検討する
