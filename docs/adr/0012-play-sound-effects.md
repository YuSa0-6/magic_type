# 0012. プレイ中の効果音は Web Audio で合成し自分の操作起点のみ鳴らす

- ステータス: 承認
- 日付: 2026-06-15
- 関連: ADR 0002(純 TS エンジン分離 — 音は web 層のみでエンジンに入れない)、
  ADR 0006(web のディレクトリ層 — 音モジュールは lib に置く)、
  ADR 0007(クールダウン中の先行入力 — ドレイン受理時に音を鳴らす)、
  ADR 0008(状態同期の時間軸/入力軸分割 — 音の発火は rAF を使わずイベント/tick 駆動)。

## 文脈

プレイ中の手応えを上げるため、打鍵・ミス・カード発動などに効果音を付けたい。
音源・対象・タイミング・配線について以下を検討した。

音源:

- **A. Web Audio API でコード合成**: 周波数/波形/減衰をコードで作る。バンドル 0KB。
- **B. 録音ファイル**: クオリティは出しやすいが、アセットのバンドル増・読み込み・配布が要る。

鳴らす対象:

- **A. 自分の操作起点のみ**: 自陣の打鍵/選択/発動だけ。相手(対戦相手/ボット)は無音。
- **B. 全イベント**: 相手の発動なども鳴らす。情報量は増えるが、相手の盤外情報が音で漏れ、
  オンラインでは権威 push のたびに鳴って騒がしく、嘘の音(予測ズレ・reconcile)も出やすい。

クールダウン明けの先行入力(ADR 0007)の扱い:

- 打った瞬間に鳴らすと、正誤がまだ確定していない(クールダウン中はバッファに積むだけ)ため
  「嘘の音」になる。ドレインで実際に受理された時点で鳴らせば正誤が確定済みで嘘がない。
- ただし `drainTypeahead` は従来 `boolean`(BattleEngine)/ `Activation|null`(PlayerSide)を
  返すだけで、「何文字どの結果で受理したか」を web から観測できなかった。

配線(web 側でどう発火させるか):

- **A. エンジンの drain 戻り値を結果列に拡張**: 受理した PressResult を順序通り返し、web は
  それを音へ写す。観測が忠実。
- **B. スナップショット差分から推測**: typedRomaji の伸びなどから何が起きたか web で推測する。
  脆く、ミス/発動の区別が曖昧。

## 決定

**Web Audio 合成(音源 A)・自分の操作起点のみ(対象 A)・エンジン drain 戻り値の結果列化
(配線 A)を採用する。** 音は web 層のみに置き、エンジンには入れない(ADR 0002)。

### 1. 音源は合成・素材差し替え可能な構造

- 打鍵=短いサイン波クリック / エラー=低めの矩形短ブザー / 選択=より高い極短サイン /
  発動=上昇 3 音アルペジオ(C-E-G のサイン)。マスターゲインは控えめ(耳障りにしない)。
- 各音は小さな合成関数に閉じており、将来録音素材へ差し替える場合は中身だけ変えればよい。

### 2. 鳴らすイベントと音の対応(PressResult 起点)

- `accepted`(正常打鍵)→ 打鍵音 / `mistyped`(誤入力)→ エラー音 /
  `activated`(発動)→ 発動音(全カード共通 1 種)/ `blocked`(無効打)→ 無音 /
  `buffered`(先行入力を積んだ瞬間)→ **無音**。
- カード選択(構え)→ 控えめな選択音。実際に選択が変わった時だけ(同カード再選択・終了後の
  no-op では鳴らさない)。web 側で選択前後の selectedIndex を比較して判定する。
- 先行入力は「打った瞬間は無音」、クールダウン明けのドレインで実際に受理されたぶんだけ鳴らす
  (正誤確定後に鳴るので嘘がない, ADR 0007)。

### 3. エンジン小改修(純粋なまま・ADR 0002 維持)

- `BattleEngine.drainTypeahead(atMs)` の戻り値を `boolean` → `PressResult[]` に変更
  (受理した各打鍵の結果を順序通り返す。空なら `[]`)。`PlayerSide.drainTypeahead` は
  `{ results: PressResult[]; activation: Activation|null }` に拡張(per-key 結果と発動を両方持つ)。
  `MatchEngine.drainTypeahead` は `PressResult[]` を返す(発動ダメージ適用は従来どおり内部で行う)。
- 空配列は truthy なので、従来 `if(changed)` 等で真偽を見ていた呼び出し側は **`.length`** で
  判定するよう揃える(match-bot の `changed` など)。
- `pressKey` 内部冒頭の drain 結果は破棄される。そのままだと「直接打鍵と同 tick 内でドレイン
  された先行入力」の音が失われるため、**web 側は各打鍵処理で先に `drainTypeahead(now)` を明示
  的に呼んで結果の音を鳴らし、その後 `pressKey(key, now)` を呼ぶ**。明示ドレインでバッファが
  空になり `pressKey` 内部のドレインは no-op になるので二重音は起きない。順序(ドレイン→当該
  キー適用)は元の `pressKey` 内部と同一なので挙動は不変。
- B(差分推測)を採らない理由: ミス/発動の区別が曖昧で脆く、嘘の音が出やすいため。

### 4. 配線(ADR 0006: ビューは薄く)

- ソロ(Game.svelte)・対ボット(Match.svelte)・オンライン(Room.svelte + SelfPredictor)の
  自陣の打鍵/ドレイン/選択にのみ音を配線する。相手側ドレイン・相手の発動は鳴らさない。
- オンラインは **自陣のローカル入力イベント時のみ**鳴らす。サーバースナップショット適用
  (reconcile / 権威 push)では鳴らさない(音は予測 MatchEngine への入力時だけ)。SelfPredictor の
  `press`/`drain` を PressResult / PressResult[] 返却に拡張してミス/発動を区別する。
- 音システムはバトル開始のユーザージェスチャ(スペース/クリック/ロビーのボタン)で resume し、
  ブラウザの自動再生制約を回避する。オンラインの matchStart はサーバー駆動でジェスチャを伴わない
  ため、ロビーの「部屋を作る」「参加する」ボタンで unlock する。

### 5. ミュート・永続化・配置

- ミュートはトグル 1 個。localStorage(`magic:sound-muted`)に保存し次回維持、既定は音オン。
  永続化は deck-storage.ts と同じ localStorage パターン(try/catch で例外を握り既定へ倒す)。
- 音モジュールは状態(ミュート)を持つアプリ全体 1 つの再生器なので、router.svelte.ts と同型の
  「モジュール singleton + $state + getter/名前付き関数 export」で lib に置く(sound.svelte.ts)。
- ミュートトグル UI はバトル画面の隅(BattleScreen / MatchBattleScreen)に小さなボタンとして置く。
  ビューは `muted` と `onToggleMute` を props で受ける薄い形にし、状態の読み書きは親が sound
  モジュール経由で行う(ADR 0002)。トグル後はボタンを blur して以後の打鍵を妨げない。

### 6. 盤面結果(被弾 / 防御 / 命中)— スナップショット差分で鳴らす(追記)

打鍵起点の音(2)に加え、盤面の結果にも音を付ける。これは「自分の操作起点のみ」(対象 A)を
意図的に拡張するものだが、対象は次の 3 つに限り、相手の盤外情報は漏らさない。

- 被弾(damage taken): 自陣 HP が減少 → 重い低音の衝撃音。被弾は相手起点だが「自陣に関わる
  結果」なので鳴らす。対戦(対ボット/オンライン)のみ(ソロは自陣 HP が無い)。
- 防御(shield block): 自陣 shield が減少 かつ HP が不変 → 金属的な短い「カキン」。対戦のみ。
- 命中(hit landed): 相手/的の HP が減少 → 軽い着弾音。全モード(ソロは的 HP の減少=自分の
  攻撃命中)。

検出は打鍵結果ではなく**権威スナップショットの HP / シールド差分**で行う。配線側は各モードで
前値(plain let の null 番兵・非リアクティブ)を持ち、スナップショットを取り直した直後に前値/
新値を音 API に渡す。前値が null の最初の観測はベースライン設定のみで鳴らさない(初期満タンや
取りこぼしでの誤発火を防ぐ)。新マッチ/再戦/再接続の初期化で前値を null に戻す。

優先規則(playSelfDamage / playEnemyHit に集約):

- HP が減れば被弾音(同時に shield も減っていても被弾優先)。
- HP 不変で shield だけ減れば防御音。
- 相手側は HP 減少のみで命中音(相手のシールドのみ減少した場合は無音=今回のスコープ外)。

オンライン(Room.svelte)でも HP/シールドは全て権威スナップショット(予測しない, ADR 0011 #9)
由来。盤面結果 $effect の依存は displaySnapshot ではなく transport.authState(=HP/シールドの
唯一の権威源)に張る。予測(打鍵)では authState が変わらず effect は再走しないので、予測二重
発火が構造的に無い。再接続(matchResumed)時は、matchResumed と直後の現況 state がサーバーから
別フレームで届く一方、authState は切断前の古い HP のまま残る(クリアしない)。displaySnapshot に
依存すると、initPredictor の prediction 再代入で走る中間 flush が「切断前の古い HP」をベース
ラインに焼き付け、続く現況 state(切断中に削られた HP)との差分で偽の被弾/命中音が鳴ってしまう。
authState 依存にすれば prediction 再代入では再走せず、matchResumed 後の最初の権威 state(=
再開後の現在 HP)が初回観測=ベースラインとなり、切断中に進んだ差分は鳴らない。

エンジンは一切変更しない(web 層のみ・ADR 0002 維持)。音は引き続きミュート尊重・AudioContext
が running の時だけ鳴る(既存ガードを使う)。各音は playDamageTaken / playShieldBlock /
playHitLanded の合成関数に閉じ、周波数スライドは toneSlide ヘルパーで作る(録音差し替えは中身
だけ変えればよい)。

## 結果

### 良い点

- 録音アセット 0KB で手応えが出る。素材差し替えは合成関数の中身だけで済む構造。
- 自分の操作起点のみなので、相手の盤外情報が音で漏れず、オンラインでも騒がしくならない。
- 先行入力は受理確定後に鳴るので嘘の音が出ない(ADR 0007 と整合)。
- drain 戻り値の結果列化で、クールダウン明けに受理されたぶんを忠実に鳴らせる。
- 音は web 層のみでエンジンは純粋なまま(ADR 0002)。決定論・テスト容易性を損なわない。

### トレードオフ

- `drainTypeahead` の戻り値型が変わり、真偽で見ていた呼び出し側を `.length` 判定へ追従させる
  必要がある(空配列が truthy なため)。
- web 側で「明示ドレイン→pressKey」の順序を守る配線規約が増える(守らないと同 tick ドレイン分の
  音が欠ける)。挙動自体は不変だが、規約として配線側が知っている必要がある。
- 合成音のクオリティは録音に劣る(本決定では手応えと 0KB を優先)。
- AudioContext の resume はユーザージェスチャ必須で、起動前のイベントは無音になる(設計上許容)。
