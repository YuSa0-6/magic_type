# 0011. 対戦(PvP)のネットコードとアーキテクチャ

- ステータス: 承認(設計決定。実装は後続 PR で段階的に行う)
- 日付: 2026-06-14
- 関連: ADR 0009(PvP 前方設計)を具体化、ADR 0003(Hono + Cloudflare Workers)、
  ADR 0004(サーバー DDD)、ADR 0005(engine の所有権)、
  ADR 0008(時間軸/入力軸分割)、ADR 0010(対戦ルール)と対をなす

## 文脈

対戦のスコープとして「オンライン対戦を直接作る」(ボット先行や非同期ゴーストを
経由しない)を選んだ。ADR 0010 で効果が「生きた相手」に干渉するゲームに決まったため、
相手の録画を再生する非同期ゴーストは原理的に成立しない(録画相手にこちらの妨害は
通らない)。よって ADR 0009 が予約した足場を実際に建設する。

このジャンルで強制される(選べない)前提:

- **サーバー権威は必須**。HP・相手への効果適用・時間切れ判定・同時撃破の裁定・
  アンチチートを、クライアントの自己申告に委ねられない(ADR 0004 の
  「サーバー権威の判定」)。
- **自陣のローカル予測は必須**。タイピングゲームで自分の打鍵フィードバックを
  RTT 待ちにすると操作不能になる。

したがって「サーバー権威 + 自陣予測 + 敵陣はサーバープッシュ」は確定し、本当の
設計判断は「クライアントが何を送り、サーバーが打鍵にどこまで権威を持つか」と
「マッチング・識別・切断」だった。

検討した入力粒度: ロックステップ / 発動イベント権威 / 打鍵ストリーム権威。

## 決定

### 権威と入力

1. **サーバー権威 + 自陣ローカル予測 + 敵陣サーバープッシュ**。
2. **入力粒度 = 打鍵ストリーム権威**。クライアントは全 keydown を
   `{ handIndex, key, atMs }` のコマンドとして送信し、サーバーが `TypingSession` を
   権威実行する。
   - 上り: 打鍵を約 30〜50ms フレームでバッチ送信し、Durable Object の起床回数を抑える。
   - 下り: 相手の状態は全打鍵ではなく、意味のある変化(詠唱開始・進捗・発動・効果・HP)を
     約 10Hz(ADR 0008 の表示解像度)でデルタ送信する。相手の個別打鍵は送らない。
   - 和解: 全状態の送り直しではなくデルタ/イベント差分を送り、クライアントは未 ACK の
     自分の入力を再適用する(input replay、ADR 0009 #4 の serialize/restore + replay)。
   - 理由: 「性能最優先」は「送信量最小」ではなく低遅延であり、自陣の体感遅延は
     ローカル予測で既にゼロ。よって粒度はアンチチートとミリ秒精度の先撃ち裁定で選ぶ。
     打鍵ストリームは ADR 0009 #3 のコマンドログ・決定論不変条件にそのまま乗り、
     打鍵頻度(< 20 バイト × 約 10 打/秒)のコストは無視できる。
   - なお決定論でも和解はゼロにならない。相手の妨害(例: `slow` で自分の CD 延長)は
     予測不能な外部入力なので、サーバープッシュで届いて自陣のローカル予測を補正する。
     これが和解の主戦場となる。

### エンジンの一般化

3. **`BattleEngine`(的 = targetHp)を 2 陣営の `MatchEngine` へ一般化する**
   (ADR 0009 #5)。各陣営を `playerState { hp, casting, cooldown, activeEffects }` で
   表し、権威時刻(`serverTick` / `authoritativeAtMs`)を持たせる。効果は MatchEngine が
   両陣営をまたいで適用する(ADR 0010 の `Effect` ハンドラ)。ソロモードは
   「行動しない敵」を同型で表す。
4. **コマンドログと serialize/restore を実装する**(ADR 0009 #3/#4)。受理打鍵を
   完全記録し、`from-snapshot + replay(commands)` で任意 tick を再構成できるようにする。
   これが和解と切断再開の土台になる。現状の `eventLog`(統計用)とは責務を分離する。

### マッチング・識別・配置

5. **1 マッチ = 1 Durable Object**。DO が両陣営の権威エンジンと権威時刻を保持する。
6. **マッチング = ルームコード(プライベート)**。一方が部屋を作りコードを発行、
   もう一方がコードで参加する。マッチングキューは作らない(クイックマッチは将来)。
   プレイヤー人口ゼロでも 2 タブ/フレンドで動作確認できる。
7. **識別 = エフェメラル ID**。永続アカウントは作らず、サーバー発行の使い捨て ID で
   セッション中だけ識別する。デッキは localStorage に保存し、マッチ開始時にサーバーへ
   送信、サーバーが合法性(20 枚・同種最大 2・実在カード)を検証してから権威の
   山札 seed を生成する。
8. **切断 = 猶予つき再接続**。切断後 約 30 秒は DO が権威状態を保持して試合を一時停止し、
   エフェメラル ID + マッチ ID で再接続を許す(serialize/restore が土台)。猶予を
   超過したら切断側の放棄負けとする。

### 層の配置(ADR 0004 を踏襲)

9. WebSocket / Durable Objects / ルーム管理は `apps/server` のサーバー固有ロジックと
   して `domain`(純 TS の MatchEngine・効果)・`lib`(DO/ストレージのラッパー)・
   `routes`(WS エンドポイント)に分けて配置する。判定の権威は引き続き engine を
   唯一の正とし(ADR 0005)、ロジックを二重化しない。自陣のローカル予測は、サーバーと
   同一の純 TS エンジン(ADR 0002/0005)を web 側でも実行する(権威はサーバー、予測は
   同一コードのローカル実行)。これで「同じ入力列 + atMs 列 + seed → 同じ状態」が予測と
   和解の両方で成立する。

### 時間・一時停止・結果軸(設計レビュー反映で確定)

PR レビューで洗い出したネットコード側の未定義事項を以下に確定する。

10. **終了判定専用の権威タイマ**。時間切れの勝敗確定は、どちらのプレイヤーも打鍵しない時間軸上でも
    起きる権威イベントである。純イベント駆動(打鍵時のみ評価)では「両者が打鍵を止めると試合が
    終わらない」ため、DO の alarm で制限時間 deadline に terminal イベントを注入する。効果失効は
    自律的な副作用を持たない(次回 CD などを読むときに効くだけ)ので alarm は不要で、参照時の
    遅延評価(ADR 0010 #4)で足りる。よって alarm が保持する deadline は制限時間の 1 点のみ。これは
    ADR 0008 が撤廃した「毎フレーム再計算ループ」とは別物の、表示用 tick とも独立した単発
    スケジュールである。
11. **一時停止中は権威時計を凍結する**。再接続猶予(#8)の間は権威時計を止め、制限時間・effect の
    失効・`castTimeMs`・`cooldownUntilMs` のすべてを停止ぶんオフセットする。凍結しないと切断側が
    一時停止で自分の CD 回復や相手 haste の失効を稼ぐ悪用が成立する。オフセットは決定論
    replay(#4)でも再現する。再開時は #10 の alarm も停止ぶん再スケジュールする(凍結中に
    旧 deadline で誤発火し試合を誤終了させないため)。
12. **結果軸の語彙一般化**。ソロ専用の `finished: boolean` / `clearTimeMs` を、
    `outcome: 'win' | 'lose' | 'draw' | 'forfeit'` と `endReason`(撃破 / 時間切れ / 放棄)へ
    一般化する(ADR 0009 #5 の snapshot 一般化の結果軸を補完)。裁定規則は ADR 0010 #16。
13. **rng は陣営ごと独立ストリーム**。単一 seed の逐次消費だと、自陣でローカル予測する sift の
    draw と、サーバー権威順で確定する相手起因の draw が混ざって desync する。共有マスター seed
    (ADR 0009 #2)から陣営ごとに決定論的に派生して独立ストリーム化し、各陣営の山札 rng はその
    陣営の権威ストリームでのみ消費する
    (相手起因の discard 補充も victim 側ストリーム)。これで from-snapshot + replay(#4)と予測が
    一致し、相手の引き順も読めない。

## 結果

### 良い点

- ADR 0009 が予約した足場(決定論・seed・コマンドログ・serialize/restore・
  snapshot 一般化)が実際の決定として確定し、実装 PR の土台になる。
- 打鍵ストリーム権威により完全なアンチチート(打っていない詠唱は発動できない)と
  ミリ秒精度の先撃ち裁定が得られ、ADR 0010 の効果(相手への妨害)を公平に解決できる。
- 自陣ローカル予測で自分の打鍵遅延はゼロのまま、相手干渉のみを和解対象に絞れる。
- ルームコード + エフェメラル ID で、アカウント基盤やマッチングキューを先払いせずに
  最小構成でオンライン対戦を出せる。

### トレードオフ

- 打鍵ストリームは DO の起床頻度を上げる(バッチで緩和)。サーバーが TypingSession を
  毎打鍵実行するため、ソロより CPU を使う(打鍵処理は O(1) で実害は小さい)。
- `BattleEngine` → `MatchEngine` の一般化で公開境界(snapshot 語彙・型)が変わり、
  ADR 0008 で増えた 2 系統 API(timers/state)を再度触る。web 側の利用箇所に波及する
  (ADR 0005 のレビュー運用で確認)。
- エフェメラル ID はクロスデバイス・永続ランキング・フレンド再戦を持たない。それらが
  必要になればアカウント制を別 ADR で検討する。
- ルームコードのみのためマッチング体験は限定的(クイックマッチは将来の追加)。
