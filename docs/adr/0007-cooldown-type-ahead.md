# 0007. クールダウン中の先行入力(type-ahead)を受け付ける

- ステータス: 廃止(ADR 0013 により撤回)
- 日付: 2026-06-14
- 関連: ADR 0001(正確性を発動条件としたダメージ)、ADR 0008(状態同期の時間軸/入力軸分割)
  — 先行入力のドレイン契機は ADR 0008 の時間 tick に依存する。

## 文脈

ベース実装では、クールダウン中の打鍵は `pressKey` が無条件に `'blocked'` を返して
捨てていた(`battle.ts`、`battle.test.ts` が挙動をロック)。CONTEXT.md も当初
「クールダウン中もカード選択(構え)は可能だが、打鍵が受理されるのはクールダウン
明けから」と定義し、先行入力を捨てることを正としていた。

しかしパフォーマンス/体感レビューで、速いプレイヤーは発動→クールダウン中に次の
カードを構えて先行入力するため、クールダウン明け先頭の1〜数打鍵を毎発動ごとに
落とすことが判明した。タイムアタックで「取りこぼし感」の主因になっている。

検討した案:

- **A. 先行入力バッファ**: 構え済みのクールダウン中打鍵をキューし、明けにまとめて受理
- **B. 現状維持**: 打鍵は受理しない(最小実装だが体感問題が残る)
- **C. クールダウン中も詠唱を完全進行**: 打鍵を即受理し、ダメージ適用だけ明けへ遅延

## 決定

**A を採用する。**

- 構え済み(`selectedIndex`/`session` が有効)のカードに対するクールダウン中の打鍵を
  バッファし、クールダウン明けに順に `acceptKey` へ流す。
- **受理時刻はクールダウン明けの時刻で統一する**(`castStartedAtMs` もそこから)。
  元の打鍵時刻は使わない。これにより詠唱時間(最初の受理打鍵から発動まで、ADR 0001)が
  クールダウン中まで遡って壊れることを防ぎ、先行入力しても詠唱時間が不当に伸びない。
- 別カードへ構え直すとバッファは進捗ごとリセットする(カード切り替え=進捗リセットと整合)。
- **ドレイン契機は ADR 0008 の時間 tick** とし、エンジンに内部スケジューラは持たせない
  (時刻外部注入の決定論設計を維持)。ドレインで入力軸の状態が変化した場合は、
  UI 側で入力軸スナップショットも更新する。
- バッファには上限を設ける(暴走防止。クールダウンは短く人間の打鍵数は限られるため小さい値で足りる)。

C を採らない理由: クールダウン中に詠唱を完了・スタックできてしまい、クールダウンが
スループットを律速するという現行のゲームバランスを崩すため(レビューでも危険と指摘)。

## 結果

### 良い点

- クールダウン明けの取りこぼしが解消し、上級者の体感とタイムが改善する。
- C と違いクールダウンのスループット律速は維持され、ゲームバランスを崩さない。
- 受理時刻をクールダウン明けに統一するため、詠唱時間の意味(ADR 0001)が保たれる。

### トレードオフ

- バッファ上限・リセット規則という新しい状態がエンジンに加わる。
- ドレインが tick 頻度(約 100ms)に律速され、クールダウン明け「即時」ではない
  (人間の知覚では十分だが、厳密には数十ミリ秒の遅延がありうる)。
- 「クールダウン中に詠唱を終えてスタックしておく」はできない(C との差。意図的な制約)。
