# 0002. ドメインロジックをフレームワーク非依存の純TSエンジンに分離する

- ステータス: 承認
- 日付: 2026-06-10
- 補記(2026-06-12): ADR 0004 により、engine の役割は「タイピング/バトルの
  エンジン」から「製品全体の唯一のドメイン層」へ拡張された。サーバー発の概念
  (ルーム・マッチング等)も engine にモデリングする。本 ADR の決定(純TS・
  フレームワーク非依存・UIとの境界は入力イベントと状態スナップショット)は
  引き続き有効。

## 文脈

UIはSvelteで作る。Svelteのrunesにバトルロジックを直接書けば書き味は最速で、
タイムアタック規模なら十分動く。一方、本作のドメインロジック
（ローマ字判定ステートマシン・詠唱状態・ダメージ計算・デッキ/手札・クールダウン）
には次の性質がある:

- ローマ字判定は本作の品質リスクの核であり、UIなしで網羅的にテストしたい。
- 将来の対戦モードでは、クライアントの自己申告を信用せず、サーバ側で同じ
  判定ロジックを走らせて権威とする可能性が高い。
- タイムアタックの計時やリプレイ検証など、UIと無関係な機能が今後も増える。

## 決定

ドメインロジックを `src/engine/` にフレームワーク非依存の純TypeScriptとして
分離する。バトルを集約としてエンジン側が一元管理し、UI(Svelte)は次の責務に
限定する:

- キー入力・カード選択操作をエンジンに渡す
- エンジンの状態スナップショットを表示に反映する
- UI固有の状態(ハイライト、アニメーション等)のみを自前で持つ

境界をまたぐのは「入力イベント」と「状態スナップショット」のみとし、
業務ルール(ダメージ・詠唱・クールダウン等)をUI層に書かない。

## 結果

### 良い点

- エンジンをVitestでUIなしに丸ごとテストできる(表記ゆれの網羅検証を含む)。
- 対戦モード実装時、同じエンジンをサーバへ持ち込んで判定の権威にできる。
- UIフレームワークの差し替え・リプレイ機能などに対してロジックが無傷で済む。

### トレードオフ

- 小規模ゲームとしては層が一枚増え、状態をストアへ橋渡しする定型コードが
  必要になる。Svelte直書きに比べ初速は落ちる。
- エンジンのAPI(入力イベントと状態スナップショットの形)を意識的に設計する
  必要がある。
